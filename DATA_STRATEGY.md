# 메디테치 바디맵 - 데이터 전략 문서

## 1. 개요

메디테치 바디맵은 **단일 데이터 소스에 의존하지 않고, 다양한 공공데이터 API와 AI 서비스를 조합**하여 최상의 결과를 도출합니다.

> **상세 API 연동 명세**: 각 API의 엔드포인트, 요청/응답 필드, 데이터 가공 파이프라인, 응급 모드 설계 등은 [API_INTEGRATION_PLAN.md](./API_INTEGRATION_PLAN.md)를 참고하세요.

핵심 원칙:
- 공공데이터 API를 기본 데이터 소스로 활용
- 여러 소스의 데이터를 교차 검증하여 정확도 향상
- AI(LLM)를 활용하여 데이터 분석 및 자연어 처리
- 사용자 피드백을 반영하여 지속적 개선

## 2. 데이터 소스 분류

### 2.1 공공데이터 API

| API | 제공 기관 | 주요 데이터 | 활용 기능 |
|-----|-----------|-------------|-----------|
| 건강보험심사평가원 API | HIRA | 병원 기본정보, 진료과목, 운영시간, 평가등급 | 병원/약국 탐색, 진료과 매칭 |
| 공공데이터포털 (data.go.kr) | 행정안전부 | 전국 의료기관 현황, 약국 정보, 의약품 허가 정보 | 병원/약국 검색, 약 정보 조회 |
| 의약품안전나라 (nedrug.mfds.go.kr) | 식약처 | 의약품 안전사용 정보, 복약 안내, 부작용 정보 | 진료 후 설명 재구성, 복약 안내 |
| 응급의료포털 API | 중앙응급의료센터 | 응급실 실시간 병상 정보, 대기 현황 | 응급 상황 안내 (향후 확장) |

### 2.2 외부 서비스 API

| API | 용도 | 활용 기능 |
|-----|------|-----------|
| Kakao Map API | 위치 기반 검색, 거리 계산, 길찾기 | 병원/약국 탐색, 길찾기 연결 |
| Naver Map API | 위치 검색 보조 (대체 또는 보조 소스) | 교차 검증용 |
| OpenAI API (GPT) | 자연어 처리, 텍스트 분석 | 증상 분석, 설명 재구성, 채팅 |
| Claude API | 자연어 처리 대안 | LLM 제공자 전환 시 활용 |

### 2.3 내부 데이터 (Supabase)

| 테이블 | 데이터 | 용도 |
|--------|--------|------|
| users | 사용자 정보 | 인증, 개인화 |
| symptoms | 증상 입력 기록 | 증상 분석, 이력 관리 |
| chat_logs | 채팅 기록 | 대화 맥락 유지, 분석 개선 |
| api_cache | API 응답 캐시 | 응답 속도 개선, API 호출 절약 |

## 3. 데이터 통합 파이프라인

### 3.1 아키텍처

```
[사용자 입력]
     ↓
[백엔드 데이터 통합 레이어]
     ├── 증상 분석 파이프라인
     │    ├── 사용자 입력 구조화 (body_part, pain_level, pain_type...)
     │    ├── LLM API 호출 → 증상 해석 + 진료과 매칭
     │    └── 내부 증상-진료과 매핑 데이터 교차 검증
     │
     ├── 병원/약국 탐색 파이프라인
     │    ├── 건강보험심사평가원 API → 병원 기본 정보
     │    ├── 공공데이터포털 API → 추가 정보 보강
     │    ├── 지도 API → 위치/거리/길찾기
     │    └── 결과 병합 + 정렬 (거리, 평가, 운영 여부)
     │
     ├── 약 정보 파이프라인
     │    ├── 의약품안전나라 API → 약 안전 정보
     │    ├── 공공데이터포털 API → 약 허가 정보
     │    └── LLM API → 쉬운 말로 요약
     │
     └── 진료 후 설명 파이프라인
          ├── 사용자 입력 (의사 설명, 처방, 주의사항)
          ├── LLM API → 쉬운 말 재구성
          ├── 약 정보 API → 처방약 상세 정보 보강
          └── 구조화된 결과 출력 (해야 할 것, 주의점, 질문 제안)
```

### 3.2 결과 도출 방식

각 파이프라인은 다음 원칙으로 최상의 결과를 도출합니다:

1. **다중 소스 조합**: 단일 API 응답에 의존하지 않고, 가능한 여러 소스를 조합
2. **우선순위 기반 병합**: 데이터 신뢰도와 최신성을 기준으로 우선순위를 부여
3. **캐싱 전략**: 자주 조회되는 데이터는 캐시하여 응답 속도 개선
4. **폴백 처리**: 특정 API 장애 시 대체 소스로 자동 전환
5. **AI 보강**: 원시 데이터를 LLM으로 분석하여 의미 있는 결과로 변환

## 4. 단계별 고도화 계획

### 4.1 1단계 (MVP)

목표: 기본 기능 동작 확인

- 공공데이터 API 1~2개 연동 (건강보험심사평가원, 의약품안전나라)
- OpenAI API로 증상 분석 및 설명 재구성
- Kakao Map API로 병원/약국 위치 검색
- 병원/약국 정보는 목업 데이터로 시작, 점진적으로 실데이터 전환

### 4.2 2단계 (데이터 확장)

목표: 데이터 정확도 및 커버리지 향상

- 공공데이터포털 API 추가 연동
- 여러 API 응답을 교차 비교하여 정확도 향상
- API 응답 캐싱 레이어 구현 (Redis 또는 Supabase 활용)
- 증상-진료과 매핑 데이터 자체 구축 시작

### 4.3 3단계 (고도화)

목표: 개인화 및 추천 정확도 극대화

- 사용자 이용 패턴 분석 반영
- 병원 평가/리뷰 데이터 통합
- 지역별 의료 접근성 데이터 반영
- 추천 알고리즘 고도화 (단순 매칭 → 가중치 기반)
- 다국어 지원 (영어, 중국어 등)

## 5. API 키 관리

모든 API 키는 환경변수로 관리하며, 코드에 직접 포함하지 않습니다.

```bash
# .env 파일 구성
DATA_GO_KR_API_KEY=                 # 공공데이터포털 통합 인증키 (모든 공공 API 공용)
NEXT_PUBLIC_KAKAO_MAP_API_KEY=      # Kakao Map JavaScript API 키 (프론트엔드 노출용)
OPENAI_API_KEY=                     # OpenAI API 키
```

### API 키 발급 방법

1. **공공데이터포털**: https://www.data.go.kr 회원가입 → 원하는 API 활용 신청 → 인증키 발급
   - 건강보험심사평가원 (병원정보, 상세정보, 약국정보, 병원코드)
   - 식품의약품안전처 (DUR 품목정보, DUR 성분정보)
   - 국립중앙의료원 (응급의료기관 정보)
   - 의료기관평가인증원 (인증현황)
   - 위 API 모두 공공데이터포털 하나에서 신청 가능, 동일 인증키(ServiceKey) 사용
2. **Kakao Map**: https://developers.kakao.com → 앱 생성 → JavaScript 키 발급 (지도 표시, 길찾기)
3. **OpenAI**: https://platform.openai.com → API 키 발급

## 6. 데이터 품질 관리

### 6.1 데이터 검증 원칙

- API 응답의 필수 필드 존재 여부 확인
- 비정상적인 값 (음수 좌표, 빈 문자열 등) 필터링
- 동일 병원에 대한 다중 소스 정보 불일치 시 최신 데이터 우선

### 6.2 사용자 피드백 반영

- "이 정보가 도움이 되었나요?" 피드백 수집 (향후)
- 잘못된 정보 신고 기능 (향후)
- 피드백 데이터를 기반으로 추천 가중치 조정

## 7. 면책 사항

본 서비스는 **의료 진단 서비스가 아닙니다**. 제공되는 모든 정보는 참고용이며, 정확한 진단과 치료는 반드시 의료 전문가와 상담해야 합니다. 공공데이터 API의 정보는 업데이트 주기에 따라 실제와 차이가 있을 수 있습니다.
